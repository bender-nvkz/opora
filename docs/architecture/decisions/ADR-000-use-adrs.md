# ADR-000: Использование Architecture Decision Records

**Date:** 2026-05-01
**Status:** Accepted
**Deciders:** Core team

## Context

Платформа Опора — долгосрочный open-source проект с несколькими разработчиками
и AI-агентами в рабочем процессе. Архитектурные решения должны быть задокументированы
так, чтобы их мог понять как новый разработчик, так и AI-агент с контекстным окном.

Без явной документации решений:
- Причины выбора технологий теряются после ухода автора.
- AI-агенты принимают решения, противоречащие уже принятым.
- Ревью PR затруднено — неясно, нарушает ли изменение намеренный выбор.

## Decision

Использовать Architecture Decision Records (ADR) по формату Майкла Найгарда
с расширениями для нашего проекта.

**Правила:**
1. Новый ADR создаётся при каждом изменении, влияющем на архитектуру системы.
2. ADR нумеруются последовательно (ADR-000, ADR-001, ...).
3. Файл размещается в `docs/architecture/decisions/`.
4. Статусы: `Proposed` → `Accepted` | `Rejected` → (при изменении) `Superseded by ADR-NNN`.
5. ADR неизменяемы: если решение изменилось — создать новый ADR, отметить старый как Superseded.
6. PR с архитектурным изменением без ADR отклоняется на ревью.

**Формат файла:**
```
# ADR-NNN: Заголовок

**Date:** YYYY-MM-DD
**Status:** Proposed | Accepted | Rejected | Superseded by ADR-NNN
**Deciders:** Список участников

## Context
[Проблема, которую решаем]

## Decision
[Что конкретно решено]

## Consequences
[Положительные и отрицательные последствия]

## Alternatives Considered
[Что рассматривали и почему отклонили]
```

## Consequences

**Положительные:**
- Решения доступны всей команде и AI-агентам в git.
- Новые разработчики понимают причины, а не только результат.
- Ревью PR с ADR значительно продуктивнее.

**Отрицательные:**
- Дополнительное время на документирование (~30 мин на ADR).
- Старые ADR могут вводить в заблуждение без актуального статуса.

## Alternatives Considered

1. **Wiki (Confluence/Notion):** Отклонено. Контекст оторван от кода, не версионируется
   с PR, не виден AI-агентам без отдельной интеграции.
2. **Обсуждения в PR:** Отклонено. PR-дискуссии недоступны после merge, плохо ищутся.
3. **Один большой техническй спек:** Отклонено. Монолитные документы быстро устаревают.
4. **RFC с голосованием:** Отклонено как избыточно тяжёлое для команды нашего размера.
