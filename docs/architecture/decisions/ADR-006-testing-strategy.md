# ADR-006: Стратегия тестирования

**Date:** 2026-05-21
**Status:** Accepted
**Deciders:** Core team

## Context

Опора — production-ready PIM/MDM платформа с Clean Architecture. Разные слои
имеют разные характеристики с точки зрения тестируемости:

- **Domain layer** (`src/Domain/`): чистый PHP, детерминированный, легко тестируется
  в изоляции. Нет внешних зависимостей кроме PSR-интерфейсов.
- **Application layer** (`src/App/`): команды, хендлеры, сервисы. Зависит от
  интерфейсов, легко мокируется.
- **Infrastructure layer** (`src/Infrastructure/`): ORM-мапперы, HTTP-адаптеры,
  консольные команды. Требует реальных зависимостей (БД, filesystem).

До ADR-006 в проекте не было формализованной стратегии тестирования —
существовали отдельные требования в `CONTRIBUTING.md` («unit-тесты для Domain
обязательны»), но не было единой методологии, привязанной к слоям архитектуры.

## Decision

Используем **стратифицированный TDD** — разные подходы для разных слоёв:

| Слой | Подход | Тип теста | Порядок |
|------|--------|-----------|---------|
| Domain (`src/Domain/`) | Строгий TDD | Unit, без моков на конкретные классы | Тест → реализация |
| Application (`src/App/`) | Specification-first | Unit, моки на интерфейсы | AI пишет тест → подтверждение → реализация |
| Infrastructure (`src/Infrastructure/`) | Implementation-first | Integration, реальная тестовая БД | Реализация → интеграционный тест |

### Domain layer — строгий TDD

Правила:
- Тест пишется **до** реализации кода
- Тест — это спецификация поведения, а не документация
- Если ты не можешь написать тест до кода — скорее всего интерфейс
  недостаточно продуман
- Моки только на интерфейсы, никогда на конкретные классы
- Нет зависимостей от фреймворка (Yii, Cycle ORM)

### Application layer — specification-first

Правила:
- Перед реализацией хендлера — фиксируется контракт тестом
- Паттерн: AI пишет тест → разработчик читает и подтверждает → AI реализует
- Команда/хендлер — это оркестрация, а не бизнес-логика; тест проверяет,
  что вызваны правильные методы репозиториев и опубликованы правильные события

### Infrastructure layer — implementation-first

Правила:
- Чистый TDD здесь механически сложен (нужна реальная БД, filesystem)
- Порядок: сначала реализация, потом интеграционный тест
- Mock-free: тестируем с реальной тестовой БД (контейнер)
- Интеграционные тесты изолированы от unit-тестов в CI

### Именование тестов

Формат: `test_[что_тестируем]_[при_каком_условии]_[ожидаемый_результат]`

```php
// ✅ Правильно
public function test_create_folder_throws_when_slug_conflicts(): void
public function test_config_raises_exception_when_debug_enabled_in_production(): void

// ❌ Неправильно
public function testCreate(): void
public function test1(): void
```

### Обязательный охват

Для каждого среза разработки:
1. Все **инварианты из спеки** (каждая спека содержит раздел с инвариантами,
   например §6 в `1_1-opora-core.md`; каждый инвариант должен быть покрыт тестом)
2. Все **граничные случаи** (slug конфликт, превышение глубины,
   disabled пользователь, пустые коллекции, null-значения)
3. **Happy path** — основной сценарий использования

### Расположение тестов

Тесты лежат рядом с кодом зеркально структуре `src/`:

```
packages/core/src/Folder/FolderService.php
packages/core/tests/Unit/Folder/FolderServiceTest.php

packages/core/src/Module/ModuleRegistry.php
packages/core/tests/Unit/Module/ModuleRegistryTest.php
```

### Запрещённые паттерны в тестах

- `@runInSeparateProcess` без явной причины и комментария
- Тесты с `sleep()` или `usleep()`
- Прямые SQL-запросы в unit-тестах (только в integration)
- Моки на конкретные классы — только на интерфейсы
- `@covers` аннотации без реального кода
- Магические числа без пояснения

### Запуск тестов

```bash
make test              # все тесты
make test-unit         # только unit
make test-integration  # только integration
make test-filter=ИмяТеста  # один тестовый класс
make stan              # PHPStan level 8
make ci                # cs-check + stan + rector-check + test
```

## Consequences

### Положительные
- Тесты в Domain всегда существуют до реализации — это предотвращает
  проектирование «от реализации»
- Specification-first в Application даёт понятные контракты до кода
- Интеграционные тесты тестируют реальное поведение, а не моки
- CI может запускать unit и integration отдельно (разное время выполнения)
- Новые разработчики понимают что и как тестировать без гадания

### Отрицательные
- Интеграционные тесты требуют PostgreSQL в тестовом окружении
- Specification-first требует дисциплины: «сначала тест, потом код» —
  легко скатиться в «сначала код»
- Нужно следить, чтобы покрытие инвариантов не падало — coverage не является
  метрикой качества, качество измеряется охватом инвариантов

### Риски
- Если забыть про TDD в Domain — появляется код без тестов, который
  сложно покрыть постфактум
- Если делать specification-first формально — тесты будут тестировать
  реализацию, а не поведение

## Alternatives Considered

### 100% TDD everywhere
Infrastructure layer слишком жёстко привязан к внешним зависимостям.
Чистый TDD там создаёт хрупкие тесты, которые тестируют моки, а не поведение.

### Только integration-тесты
Слишком медленно, невозможно тестировать Domain-логику в изоляции.
Один тест с БД — миллисекунды против микросекунд для unit.

### Только unit-тесты
Не покрывают взаимодействие с реальной БД, filesystem, внешними API.
Опасность: «всё работает в unit, падает в production».

## Связь с другими ADR

- **ADR-003** (Repository Structure): репозитории в Domain — только интерфейсы;
  реализации — в Infrastructure. Это напрямую влияет на стратегию тестирования:
  Domain-тесты мокают интерфейсы репозиториев, Infrastructure-тесты
  тестируют реализации с реальной БД.
- **ADR-005** (Module Lifecycle Contract): модули реализуют
  `ModuleInstallerInterface`; тесты установки — интеграционные (требуют БД).
