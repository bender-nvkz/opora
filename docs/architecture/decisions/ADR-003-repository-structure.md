# ADR-003: Dual-license схема репозиториев

**Date:** 2026-05-01
**Status:** Accepted
**Deciders:** Core team

## Context

Опора строится по модели open-core: Community Edition (CE) открытый и бесплатный,
Enterprise Edition (EE) — коммерческий. Нужно решить:
- Как организовать код CE и EE в git.
- Как обеспечить лицензионную чистоту.
- Как работать с GitLab CE как обязательным зеркалом для РФ-контекста.

Требования:
- CE под AGPLv3 (обязывает публиковать изменения, защищает от проприетарного forking).
- EE под проприетарной лицензией MEIJI MEDIA (закрытый код).
- Возможность развивать EE поверх CE без конфликтов.
- Зеркалирование на GitLab CE как резервная инфраструктура.

## Decision

**Два раздельных репозитория:**
- `opora/opora` (GitHub, public) — CE под AGPLv3. Содержит ядро.
- `opora/opora-enterprise` (GitHub, private) — EE. Зависит от CE как Composer-пакета.

**Правила изоляции:**
- `opora/opora` не содержит ни строчки EE-кода.
- `opora/opora-enterprise` расширяет CE через официальный Extension API (Этап 12).
- LICENSE файл в корне каждого репозитория — соответствующий лицензии.

**GitLab CE зеркало (обязательно):**
- Автоматическое зеркалирование из GitHub → GitLab CE через webhook.
- Используется как запасной источник для РФ-команды при недоступности GitHub.
- Не является первичным источником истины — только зеркало.

**Ветки:**
- `main` — trunk. Единственная долгоживущая ветка. Защищена.
- Feature-ветки: `feat/<id>-<slug>`, удаляются после merge.
- Нет `develop`, нет `release/*` — trunk-based development.

**Conventional Commits:**
- Обязательны. Используются для автоматической генерации CHANGELOG.
- Примеры: `feat(catalog):`, `fix(schema):`, `docs(adr):`, `chore(ci):`.

## Consequences

**Положительные:**
- Чёткая граница CE/EE исключает случайное смешивание лицензий.
- AGPLv3 на CE защищает от проприетарного использования без публикации изменений.
- GitLab-зеркало обеспечивает доступность кода при санкционных рисках.

**Отрицательные:**
- Поддержка двух репозиториев требует дисциплины при работе с общим кодом.
- Extension API должен быть достаточно богатым для EE-функций (реализуется в Этапе 12).

## Alternatives Considered

1. **Монорепо (CE + EE в одном):** Отклонено. Усложняет лицензирование, риск случайной
   публикации EE-кода, затрудняет open-source модель.

2. **Git submodules:** Отклонено. Сложны в работе, плохо поддерживаются инструментами.

3. **Private fork модель (EE — закрытый fork CE):** Отклонено. Делает слияние изменений
   из CE в EE ручной и болезненной операцией.
